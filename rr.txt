Prompt Codex — Smart Industrial Asset Intelligence PWA (MVP Hackathon)
Contexte général
Tu dois développer une PWA (Progressive Web App) complète pour la gestion de maintenance industrielle. Il existe deux fichiers ZIP Stitch contenant le design déjà défini — un pour l'app mobile (technicien) et un pour la plateforme web (manager/admin). Tu ne modifies pas le design. Tu l'implémentes tel quel et tu le complètes avec les fonctionnalités décrites ci-dessous.

Base de données SQLite — tables à créer
sqlCREATE TABLE users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  email TEXT UNIQUE NOT NULL,
  password_hash TEXT NOT NULL,
  role TEXT CHECK(role IN ('technicien', 'manager', 'admin')) NOT NULL,
  nom TEXT,
  prenom TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE equipements (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  nom TEXT NOT NULL,
  modele TEXT,
  numero_serie TEXT UNIQUE,
  fabricant TEXT,
  localisation TEXT,
  criticite TEXT CHECK(criticite IN ('faible', 'moyenne', 'haute', 'critique')),
  date_achat DATE,
  statut TEXT DEFAULT 'actif',
  qr_code_url TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE pannes (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  equipement_id INTEGER REFERENCES equipements(id),
  technicien_id INTEGER REFERENCES users(id),
  description TEXT,
  severite TEXT CHECK(severite IN ('mineur', 'modere', 'urgent', 'critique')),
  statut TEXT DEFAULT 'en_attente',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  closed_at DATETIME
);

CREATE TABLE photos (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  panne_id INTEGER REFERENCES pannes(id),
  url TEXT NOT NULL,
  annotation TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE workflow_etapes (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  panne_id INTEGER REFERENCES pannes(id),
  titre TEXT NOT NULL,
  assignee_id INTEGER REFERENCES users(id),
  statut TEXT DEFAULT 'en_attente',
  ordre INTEGER,
  deadline DATETIME,
  completed_at DATETIME
);

CREATE TABLE demandes_pieces (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  panne_id INTEGER REFERENCES pannes(id),
  piece_nom TEXT NOT NULL,
  quantite INTEGER DEFAULT 1,
  urgence TEXT CHECK(urgence IN ('standard', 'urgent', 'critique')),
  statut TEXT DEFAULT 'en_attente',
  commentaire TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE diagnostics_ia (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  panne_id INTEGER REFERENCES pannes(id),
  resultat_json TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

Authentification

À la connexion, le serveur génère un JWT signé avec une clé secrète locale
Le token est stocké en localStorage côté client
Chaque requête API envoie le token dans le header Authorization: Bearer <token>
Le middleware Express vérifie le token et expose req.user (id, role)
À la connexion, le frontend lit le role dans le token et redirige :

technicien → interface mobile
manager ou admin → interface web desktop




Interface 1 — App mobile (technicien)
Design fourni dans le ZIP Stitch mobile. L'implémenter exactement. Compléter avec :
Scanner QR Code

Ouvrir la caméra via html5-qrcode
Lire le QR code d'une machine → appel GET /api/equipements/:id
Afficher la fiche : nom, modèle, numéro de série, statut, historique des pannes

Signaler une panne

Formulaire : sélection machine, description, sévérité (Critique / Urgent / Modéré / Mineur)
Horodatage automatique côté serveur
Capture de 4 photos guidées avec instructions à l'écran pour chaque angle
Upload via POST /api/pannes/:id/photos

Diagnostic IA

Après upload des photos, appel à POST /api/pannes/:id/diagnostic
Le serveur envoie les photos + description à l'API Claude
Claude retourne : type de défaut probable, cause possible, pièces suggérées
Afficher le résultat dans une carte sur l'écran de la panne

Suivi de mon workflow

GET /api/pannes filtré sur technicien_id = moi
Afficher les étapes assignées avec statut
Bouton "Marquer comme fait" → PUT /api/etapes/:id/statut

Demander des pièces

Formulaire pré-rempli avec les pièces suggérées par l'IA
Le technicien confirme ou modifie
Soumission → POST /api/pannes/:id/pieces

Mode hors-ligne

Service Worker met en cache les assets et les données récentes
Formulaires remplis sans réseau stockés en IndexedDB
Sync automatique au retour du réseau

Contraintes importantes

Ne pas modifier le design Stitch — brancher les composants fonctionnels dans les conteneurs définis par le design
Utiliser les variables CSS et classes définies dans les fichiers Stitch
Le serveur Express tourne sur le port 3001, le client Vite sur le port 5173
Un fichier seed.js doit peupler la base avec des données de démonstration (5 machines, 3 users, quelques pannes) pour la démo jury
Un seul fichier .env à la racine du projet suffit :

PORT=3001
JWT_SECRET=hackathon_secret_key
ANTHROPIC_API_KEY=ta_clé_ici

L'app doit être installable sur Android et iOS (manifest.json + Service Worker configurés)
Interface mobile optimisée pour 390px, interface web pour 1280px+
